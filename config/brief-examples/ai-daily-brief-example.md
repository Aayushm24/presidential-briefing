# AI Daily Brief — Writing Style Reference

Source: The AI Daily Brief (everydayai.com). Use this as the primary style anchor for how to structure and write each section of the daily brief.

---

## What makes this newsletter work (extract before writing)

1. **Attribution after every claim.** Not at the end — immediately after the specific fact.
   Format: Person Name (Platform) Short quote or linked title
   Example: `Nufar Gaspar (LinkedIn) Nufar's full early review of Claude Design`

2. **Specific people doing specific things, not category-level summaries.**
   BAD: "Some users reported rate limit issues."
   GOOD: "Josh Gonzales hit Claude Design's separate usage limit and was locked out until the following Friday. Theo burned 10% of his usage and had a project wiped."

3. **Named verdict with specifics, not hedged analysis.**
   BAD: "The sliders are a notable feature worth exploring."
   GOOD: "The Smart Ape: 'Everyone is talking about prompting. Nobody talks about the sliders. Each one is built for your specific artifact.'"

4. **Section headers are positions, not topics.**
   BAD: "The Features"
   GOOD: "The Standout Features: The sliders are the killer feature — not the prompt box."

5. **Dense, fast-moving paragraphs.** Every sentence adds new information. No restating what you just said.

## Example excerpt (match this writing density)

> Claude Design's core value proposition, per Anthropic, is "rationing exploration" — the ability to prototype a variety of design directions before committing to one, which basically no one has time to do manually. The interaction model: prompt it to build a first version (or several), then refine using natural language, inline comments, direct canvas edits, or the per-design sliders.
>
> Two audiences stand out as the clear primary targets. First: Claude Code power users who aren't designers — the people who can build the thing but have always had to context-switch into another tool for the visual half. Second: non-designer knowledge workers, particularly marketers, who interface with design constantly but aren't practitioners.
>
> Austin Lau from Anthropic: "I described a landing page editor and Claude built me one."

**What to extract:** Every paragraph has a position + specific proof. No paragraph explains the same thing twice. Sources appear inline, not at the end.

## Example of the "what to try" section (match this format)

> Seven Things to Try
>
> (1) A slide deck — remember you don't have to hand it finished content. Give it something rough and let it reason about what to turn into slides, like an AI analysis of survey data. (2) A web project — this is going to be a core use case, complementing Claude Code for the websites and web apps you're already building. (3) A simple launch video — think SVG and word-heavy design rather than anything that needs native image generation; this is a net-new capability for most people, not a replacement.

**What to extract:** Numbered, specific, practical. Not vague "test agent workflows." Tells you exactly what to give it and why.
